About

12 years in the backend of systems that move money, handle millions of events, and can't afford downtime. Fintech, telecom, SaaS — different domains, same question: what breaks first under load?

I started my career building billing and mediation systems for telecom platforms — the kind of code where a silent bug means a customer gets billed twice and nobody notices for a week. That context stuck. I care about correctness, traceability, and failure modes more than most engineers because I've worked in domains where those things have a direct cost.
How I think about systems

Every significant system decision starts with the failure mode. Before choosing a queue vs direct call, I ask: what happens when the consumer is down? Before adding a cache, I ask: what happens on cache miss under load? This isn't pessimism — it's the only way to build systems that are debuggable at 3am.

Technical depth I've built over 12 years
  • Event-driven architectures: Kafka partition design, consumer group management, DLQ patterns, exactly-once semantics trade-offs
  • High-scale APIs: rate limiting strategies, circuit breakers, API contract versioning, backward compatibility
  • Data layer: CQRS pattern application, PostgreSQL query plan analysis, Redis cache invalidation strategies
  • Infrastructure: Kubernetes rolling/blue-green deploys, Terraform state management, AWS service selection trade-offs
  • Observability: distributed tracing, structured logging, SLO/SLI definition, alert fatigue reduction
AI in my daily workflow

I use Claude Code, Codex, and Cursor the same way I'd use a senior engineer pair — not to skip thinking, but to compress the time between thinking and working code. Practically, that means:

  • Incident root-cause: structured Claude Code chains through distributed traces cut post-incident analysis from 2 hours to 20 minutes, with a draft postmortem at the end
  • Contract test generation: feeding OpenAPI specs into Codex to scaffold xUnit contract tests before the implementation even exists
  • Migration planning: Claude Code reading a 200k-line .NET Framework codebase and producing a dependency map and breaking-change list before I touch anything
  • ADR drafts: I describe the decision context and trade-offs in conversation, get a structured ADR draft, edit it to be accurate — saves the blank-page time
  • Observability runbooks: generating alert runbooks from monitoring configs so on-call engineers have context, not just a metric threshold

The pattern is consistent: I understand the system well enough to know what to ask, and the tools compress the gap between question and answer. I still review everything. The judgment doesn't move.

The management context

I've run teams — which gives me a different read on what makes backend code good beyond correctness. Is this readable by the engineer who inherits it in 8 months? Does this ADR explain why we chose this, or just what we chose? Will this alert actually wake someone up for the right reason? Those questions change how I write code even when I'm working alone.
